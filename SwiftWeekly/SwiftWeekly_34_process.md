1) 提议[DocC 中的数学排版](https://forums.swift.org/t/pitch-mathematical-typesetting-in-docc/66418 "DocC 中的数学排版")

提出了一个新的 DocC 指令，@Math：
```Swift
/// The sample variance of the collection.
/// - Returns: The sample variance:
/// @Math("sample-variance.xml", description: "Sum, from i = 1 to n, of the squared norm of x_i minus mu. Everything divided by n minus 1.")
/// where 𝑛 is the collection's `count` and ‖𝑥ᵢ - 𝜇‖ is the Euclidean distance from each element 𝑥ᵢ to the sample mean 𝜇.
func sampleVariance …
```
**动机**
文档页面通常需要包含数学表达式。 例如，Apple 的 Accelerate 文档页面（使用 DocC 制作）通过方程式和矩阵进行了丰富，有助于澄清文档的文本。

尽管向 DocC 文档添加数学表达式已经成为可能，但当前的所有方法都还不够。

对于包含单行的表达式，例如 ‖𝑥ᵢ - 𝜇‖²，Unicode 数学通常就足够了。 但包含多行的表达式往往不可读，或者在 Unicode 数学中看起来不太好。 例如，对于上面的 TL;DR 中的表达式，我们能做的最好的事情是 1⁄₍ₙ₋₁₎ Σᵢ₌₁ⁿ ‖𝑥ᵢ - 𝜇‖²。 而且，很多时候，多行表达式不可能用 Unicode 数学表示。 例如，Unicode 中没有上标希腊字母。 矩阵也是不可能的。
我们可以将排版数学（对于 LaTeX 为 .tex，对于 MathML 为 .xml）编译为图像（例如，.png 或 .svg）。 这需要 2-3 个文件必须保持同步：浅色模式下的图像、深色模式下的图像以及可选的（但理想情况下）源文件。 这是 Accelerate 文档使用的方法（不支持深色模式）。
我们可以使用这个技巧 3，这是我个人更喜欢的，但它有自己的缺点（在链接的帖子中有详细介绍）。
因此，对数学排版的一流支持将是 DocC 的一个受欢迎的补充。 有很多不同的方法可以解决这个问题，所以我写这篇文章是为了让我们讨论潜在的解决方案。 我将包括我的个人偏好，然后是一些替代方法。

**建议的解决方案**
保持简单：编写 MathML，输出 MathML。 DocC输出一个网页； 我认为我们应该接受这一事实，并尽可能坚持网络标准。 MathML 的语法比 LaTeX 更笨拙，但是这个解决方案：

不添加依赖项，例如 在 MathJax 或 KaTeX 上。
避免在运行时将 LaTeX 编译为 SVG（或将 LaTeX 编译为 MathML，或将 MathML 编译为 SVG）的性能问题。 对于前两种情况（LaTeX 到 SVG 和 LaTeX 到 MathML），MathJax 因屏幕上有许多方程时可能导致的延迟而臭名昭著。
由于 MathML 元素使用当前字体颜色，因此无需额外工作即可适应亮/暗模式。 尝试一下：检查 DocC 网页，添加带有一些 MathML 的 <math>，然后在浅色模式和深色模式之间切换。
是最容易实施和维护的。 检查源 MathML 是否有效后，DocC 只需将其不加修改地粘贴到网页中即可。
此外，任何输出为 MathML 的方法（无论输入是 MathML 还是 LaTeX）都将具有良好的可访问性，因为用户可以使用屏幕阅读器导航 <math> 方程。 这比仅仅拥有替代文本要好，这是在 DocC 支持数学排版之前我们能做的最好的事情。 （也就是说，MathJax 对屏幕阅读器的支持也非常好。）

作为未来的方向，我们可以考虑也支持 LaTeX 作为源语言。 如果这样做，我相信我们应该将 .tex 编译为 MathML（而不是 SVG、PNG 等），以保持一致性和最佳可访问性。 此外，我们应该在编译文档时将源 LaTeX 编译为 MathML（即不是在运行时通过 MathJax 或 KaTeX），以获得更轻量的文档网页并避免性能问题。

extension Foo {

3) 讨论[处理不确定的任务执行顺序](https://forums.swift.org/t/dealing-with-non-deterministic-task-execution-order/66411 "处理不确定的任务执行顺序")
我首先会说，我意识到这个问题的范围比 TCA 更大，但我很好奇 TCA 是否可以提供解决方案。

假设我们想要写入MainActor的数据库
```Swift
actor Database {
  func insert(_ value: Int) { print("insert \(value)") }
  func delete(_ value: Int) { print("delete \(value)") }
}
```
然后我们有一个Reducer，例如，它接收来自UI的输入
```Swift
struct AppReducer: ReducerProtocol {
  struct State: Equatable { … }
  enum Action {
    case insert(Int)
    case delete(Int)
  }
  
  @Dependency(\.database) var database

  var body: some ReducerProtocol<State, Action> {
    Reduce { state, action in
      switch action {
      case let .insert(value):
        return .run { _ in
          await database.insert(value)
        }

      case let .delete(value):
        return .run { _ in
          await database.delete(value)
        }
      }
    }
  }
}
```
如果reducer按特定顺序接收操作，则由于Actor可重入性，数据库可能不会按相应顺序调用。 例如，如果减速器收到:
```Swift
viewStore.send(.insert(0))
viewStore.send(.delete(0))
```
数据库可能以错误的顺序执行，打印
```Swift
delete 0
insert 0
```
在 GCD 世界中，我们将在串行队列上执行数据库写入，但对于 Swift Concurrency，似乎没有好方法来模仿这种行为，因为每个挂起点都需要一个不确定的执行顺序。 我想知道你们是如何处理这个问题的。 我是否应该以一种根本不同的方式来思考 Swift 并发的设计模式？ 我在其他情况下也遇到过这个问题，例如，按拍摄顺序显示照片。 它适用于任何有某种输入流进入异步进程的情况，该进程产生的输出应与输入的顺序相同。

回答：
TCA 测试存储在序列化到主执行器时将表现出确定性（这是当前版本中的配置选项，也是即将发布的 1.0 版本中的默认选项）。

对于应用程序代码，如果您需要在继续之前完全处理某个操作，您可以等待它：
```Swift
await viewStore.send(.insert(0)).finish()
viewStore.send(.delete(0))  // won't execute till the above completes
```

4) 讨论[我只想编写可以在任何我想要的地方使用的扩展](https://forums.swift.org/t/i-just-want-to-write-extensions-that-i-can-make-available-wherever-i-want/66407 "我只想编写可以在任何我想要的地方使用的扩展")
提问：
“为“String”和“Int”等类型编写和使用扩展很有趣，但是当我这样做时，我的拉取请求不会获得批准，因为它们不遵守 SOLID 原则。我希望我们可以在其中编写扩展 协议，因此我们可以在继承协议的任何地方使用我们的扩展

5) 讨论[Enum Tuple case 组合/分解差异](https://forums.swift.org/t/enum-tuple-case-composing-decomposing-disparity/66406/1 "Enum Tuple case 组合/分解差异")
在以下例子中：
```Swift
typealias Tuple = (first: Int, second: Int)

enum TupleEnum {
    case tuple(Tuple)
}

func foo(value: TupleEnum) {
    switch value {
    // case let .tuple(t: Tuple): break // no brainer
    // case let .tuple((first, second)): break // no brainer
    case let .tuple(first, second): break // ✅
    }
    TupleEnum.tuple(first: 0, second: 0) // 🛑 Enum case 'tuple' expects a single parameter of type 'Tuple' (aka '(first: Int, second: Int)')
}
```
我可以使用两个单独的变量来匹配采用元组的枚举情况，但我不能执行相反的操作：用两个单独的变量构造该情况。

类似地，在这个例子中：
```Swift
typealias Tuple = (first: Int, second: Int)

enum PairEnum {
    case pair(first: Int, second: Int)
}

func bar(e: PairEnum) {
    switch e {
    // case let .pair(first, second): break // no brainer
    case let .pair(v: Tuple): break // ✅
    }
    let tuple = Tuple(first: 0, second: 0)
    PairEnum.pair(tuple) // 🛑 Enum case 'two' expects 2 separate arguments
}
```
我可以使用Tuple匹配采用两个单独变量的枚举情况，但不能执行相反的操作：用Tuple构造该情况。

我们能否以一种或另一种方式修复它并使组合和分解行为保持一致？

回答：
通过传递到Generic context，您可以这样做：
```Swift
enum PairEnum {
  case pair(first: Int, second: Int)
}

func apply<T, U>(_ f: (T)->U, _ x: T) -> U {
  return f(x)
}

let tuple = (first: 0, second: 0)
let result = apply(PairEnum.pair, tuple)    // This works!
```
由于 SE-0029，你无法直接执行此操作。

6) 讨论[为什么稍后将类型元数据转换为“AnyObject”会导致在“AnyObject”上调用“destroy_value”？](https://forums.swift.org/t/why-does-casting-type-metadata-to-anyobject-later-result-in-destroy-value-being-called-on-the-anyobject/66371 "为什么稍后将类型元数据转换为“AnyObject”会导致在“AnyObject”上调用“destroy_value”？")
给出以下代码：
```Swift
class A {
}

func b() {
  _ = A.self as AnyObject
}
```
编译器为函数 b 生成以下原始 SIL 代码：
```Swift
%0 = metatype $@thick A.Type                    // user: %1
%1 = thick_to_objc_metatype %0 : $@thick A.Type to $@objc_metatype A.Type // user: %2
%2 = objc_metatype_to_object %1 : $@objc_metatype A.Type to $AnyObject // user: %3
destroy_value %2 : $AnyObject                   // id: %3
%4 = tuple ()                                   // user: %5
return %4 : $()                                 // id: %5
```
为什么函数末尾有一个destroy_value？ 据我所知，不存在平衡的“保留”？

回答：
1. 默认情况下，AnyObject 值始终被视为保留，因此编译器必须释放它。
2. 当您将 Class 转换为 AnyObject 时，编译器可以保留它，但它选择不这样做，因为它知道 Class 是不朽的，因此不这样做可以节省代码大小和运行时间。

这两个局部推理导致了您遇到的问题。 如果释放距离 AnyObject 值的创建足够远（也许您将一个类分配给全局变量，然后分配了其他内容），则不可能对 (1) 进行更智能的处理。 因此，唯一 100% 正确的选择是在 (2) 中更加保守，以其他人不必要的操作为代价，使这个特殊的不常见用例受益。

7) 讨论[为什么是这个模式？](https://forums.swift.org/t/why-this-pattern/66392 "为什么是这个模式？")
这既涉及到一般编程，也涉及到 swift。

从 UIKit 中，我们得到：
```Swift
func touchesBegan(
    _ touches: Set<UITouch>,
    with event: UIEvent?
)
```
如果没有事件对象，这个函数几乎毫无意义。 文档并不建议您永远不会收到事件。 那么为什么这是一个可选的呢？

回答：
我不认为 UIKit 会用 nil 事件调用 TouchBegan()，但它是一个 API，您可以在其中重写一个方法，然后在您没有处理该事件时手动转发该事件，并且该转发可能无法正确转发该事件。 不幸的是，它最初允许转发 nil，并且事后很难将其更改为非 nullable。

如果今天在 Swift 中设计同样的东西，我怀疑事件参数将不是可选的。

**摘要：** 文章主要介绍了 @backDeployed 属性以及如何将函数的可用性扩展到旧的操作系统版本。 @backDeployed 属性是 Swift 5.8 首次实现的一项功能，并且在较低的最小部署目标和框架对比下，允许将新的声明提供给应用程序。文章详细解释了 @backDeployed 如何工作，并通过一个假设的 "Temperature" 类型的例子进行了演示。 @backDeployed 属性允许将函数回溯部署并使其对运行在旧版操作系统的应用程序可用。文章还指出，功能回溯部署主要对 SDK 开发者有用，同时也解释了为什么苹果不能将所有新的API回溯部署到旧的 OS 版本。使用 @backDeployed 属性进行函数回溯部署对于 SDK 开发者来说是一个很好的工具，能够使其 API 支持更有老旧的操作系统版本。虽然这不能解决所有新 API 的回溯部署问题，但它已经允许你使用一些新的、独立的 Swift API 。 

[ Swift HTTP Types 的介绍](https://www.swift.org/blog/introducing-swift-http-types/ " Swift HTTP Types 的介绍")

**摘要：**  这篇 Swift 官方博客介绍了一个名为 "Swift HTTP Types" 的新的开源软件包。该软件包旨在为 Swift 中的客户端/服务器 HTTP 操作提供共享的通用类型。
文章指出，网络化应用中的网络技术对于许多用例来说在 Swift 中是无处不在的，包括客户端、服务器、中介和其他网络参与者。在苹果平台上，系统的 HTTP 实现通过 Foundation 框架中的 URLSession API 进行暴露。而对于 Swift 服务器项目，推荐使用 SwiftNIO 实现 HTTP。
为了在 Swift 中提供最佳的 HTTP 使用体验，共享的通用类型在许多项目中至关重要。 Swift HTTP Types 提供了一种表示 HTTP 消息核心构建块的通用表现形式。它包括 HTTPRequest 和 HTTPResponse ，分别代表客户端和服务器使用的 HTTP 消息。通过在多个项目中采用这些类型，可以在不同框架之间共享更多的代码，并消除在使用多个框架时的类型转换成本。

[ Qunar 客户端 iOS 实时活动接入实践](https://juejin.cn/post/7260523399252475961?searchId=202307281123557A68223DBAE2C4960FFC/ " Qunar 客户端 iOS 实时活动接入实践")

**摘要：** 这篇文章介绍了去哪儿（Qunar）客户端如何实时接入了苹果在 iOS16 上推出的实时活动 (Live Activities) 功能。该功能允许应用在用户锁屏界面上展示实时信息，与常规推送相比，实时活动可以在一定时间段内持续展示，并于合适的时机自行消失，为用户提供更优秘体验。
通过实时活动功能，去哪儿的用户可以在出行全周期内，通过锁屏或灵动岛看到最新实时的出行信息。根据 Qunar 的数据，该功能自上线以来影响力显著，其中50%的用户已经与此功能互动，超过80%的用户因此而享受到更便捷的出行信息展示。目前，实时活动主要覆盖了航班和火车的出行场景，未来还可能延伸至船票、汽车票、景点门票等业务线场景，为 Qunar 的产品生态带来更大的贡献。
作者还详细介绍了实时活动的原理和机制，包括如何在主 APP 端启动一个实时活动实例，如何利用推送服务进行数据更新，如何关闭或更新实时活动等，并分享了在接入过程中遇到的问题及解决思路，希望帮助读者更好地理解和接入实时活动功能。。
